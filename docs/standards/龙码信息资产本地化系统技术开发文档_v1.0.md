# 龙码信息资产本地化系统技术开发文档（重写版）

## 一、文档定位

本文档不是本地知识库改造稿，不是普通 RAG（检索增强生成）工具说明，也不是单一向量库接入说明。

本文档是 **“龙码信息资产本地化系统”** 的正式技术开发母本，必须同时满足以下三个来源约束：

1. **龙码协议**：定义底层治理公理、协议对象、裁决约束、回源与审计要求。
2. **龙码信息供应链系统**：定义“原始资料接入 → 龙码信炉 → 四级知识结构 → 元数据治理 → 治理型向量知识库 → 供给 → 回源 → 审计”的完整资产生产链。
3. **本地化产品化要求**：将上述底层约束落为可独立开发、可独立部署、可独立商业化的本地私有产品。

一句话钉死：

**龙码信息资产本地化系统，是龙码协议约束下的信息供应链本地化实现，不是简单把文件搬到本地硬盘，而是把本地与外部资料转化为可治理、可回源、可审计、可供 AI 稳定调用的信息资产底座。**

---

## 二、系统在龙码体系中的正确位置

## 2.1 它是什么

龙码信息资产本地化系统是：

- 龙码协议约束下的资产底座产品
- 龙码信息供应链系统的本地化/混合化落地形态
- 面向个人与企业的私有信息资产治理系统
- 面向 AI 助手、Agent、知识问答、审查辅助、项目治理的供给底座

## 2.2 它不是什么

它不是：

- 单一向量库
- 单一文件管理器
- 单一聊天助手
- 单一大模型替代品
- 单一云盘或对象存储桶

## 2.3 它与 LIOS 的关系

必须钉死：

- **龙码信息资产本地化系统**：解决信息供给侧治理问题，负责资料吸入、炼化、沉淀、治理、供给、回源、审计。
- **LIOS**：解决更上层的 AI 系统治理、逻辑裁决、执行门控、长链路抗漂移问题。

因此：

**龙码信息资产本地化系统不是 LIOS 的附属插件。**
**它是可独立商业化的现实入口产品。**
**LIOS 是更高层的系统总装与治理平台。**

## 2.4 它与龙码协议的关系

龙码协议负责定义：

- 信息如何被治理
- 节点如何被描述
- 质量如何被计算
- 相位如何被匹配
- 证据如何被锚定
- 路径如何被裁决
- 调用如何被准入
- 结果如何被回源与审计

本系统不能绕开龙码协议另起一套字段、逻辑与裁决口径。

## 2.5 它与龙码信息供应链系统的关系

龙码信息供应链系统是上位总架构。

本系统是其本地化产品形态，必须继承以下完整闭环：

```text
原始资料接入
→ 龙码信炉
→ 四级知识结构定位
→ 元数据治理
→ 治理型向量知识库
→ 供给
→ 回源
→ 审计
→ 持续增量更新
```

## 三、系统根本目标

本系统的唯一任务，不是囤积更多资料，而是：

**把本地与外部资料持续炼成高质量信息资产，并稳定供给上层 AI 与业务系统使用。**

必须实现以下 10 个目标：

1. 承接本地与外部异构原始资料
2. 将原始资料转化为协议化资产对象
3. 建立统一四级知识结构
4. 建立热层 / 温层 / 冷层三位一体本地存储体系
5. 让信息只在被治理后进入模型视野
6. 支持回源、证据链、版本与状态追踪
7. 支持热温冷代谢与生命周期管理
8. 支持对 AI 助手、Agent、审查系统的稳定供给
9. 显著降低 AI 致幻、目标漂移、无依据输出问题
10. 成为个人版与企业版产品的共同资产底座

## 四、龙码协议对本系统的硬约束

本系统必须完整继承龙码协议的四条总公理与四层结构。

## 4.1 四条总公理的本系统映射

### 第一公理：引力定域
所有节点、知识、规则、状态、证据都必须带质量（Mass）。

本系统落地要求：
- 任何正式资产都不能平权
- 必须支持 authority_score / trust_score / reuse / recency / consistency / conflict_penalty
- 必须支持生命周期治理与热温冷迁移

### 第二公理：相位共振
候选知识、候选规则、候选动作必须与当前目标、范围、时序、主体、风险、约束保持结构一致。

本系统落地要求：
- 检索不能只看向量相似度
- 必须支持 phase_signature
- 必须在召回与供给前做作用域门控与结构匹配

### 第三公理：因果旋转
系统间传递的核心对象不是松散文本，而是状态、约束、证据锚点、路径摘要与节点对象。

本系统落地要求：
- 不允许把“自由文本”作为唯一正式产物
- 信炉必须输出协议对象、节点、锚点、摘要与索引单元

### 第四公理：时空坍塌
系统最终选择的不是“最像答案”的内容，而是“最稳、最可追溯、最不漂移”的供给路径。

本系统落地要求：
- 供给前要有质量、版本、状态、作用域、证据多维约束
- 高风险结果必须支持回源或阻断

## 4.2 四层结构对本系统的要求

### 第一层：总纲层
本系统必须有清晰定位、边界、原则与禁令。

### 第二层：协议层
本系统必须实现：
- 节点结构
- 相位结构
- 质量结构
- 状态结构
- 约束结构
- 证据锚点结构
- 审计结构

### 第三层：判定层
本系统至少要实现：
- 质量计算
- 相位/作用域匹配
- 冲突检测基础能力
- 候选精排
- 回源判断

### 第四层：执行层
本系统不是完整执行系统，但必须为上层执行系统提供：
- 可调用资产
- 可回源证据
- 可审计依据
- 可阻断的供给结果

## 七、系统的正确主链路：吸金 → 点金 → 沉金 → 供金 → 回金

为便于工程理解，本系统允许使用“金流”简称，但必须和正式架构一一对应。

## 7.1 吸金
吸入原始资料原矿。

来源包括：
- 本地文件
- 指定文件夹监听
- 指定网页
- 搜索引擎检索结果
- 指定站点抓取
- API 数据源
- 会话记录
- 录音转写结果
- 图片 OCR 结果
- 日志与代码说明

对应正式模块：**原始资料接入层**

## 7.2 点金
对原始资料进行切片、脱水、蒸发、去重、聚类、提纯。

对应正式模块：**龙码信炉中的点金坍塌**

## 7.3 沉金
把炼化结果沉淀为知识单元、摘要单元、证据单元、锚点单元、版本镜像单元、上下文包索引单元。

对应正式模块：**龙码信炉中的灵池沉淀 + 四级结构定位 + 元数据治理**

## 7.4 供金
将治理后的资产供给上层系统与模型使用。

对应正式模块：**治理型向量知识库 + 供给层**

## 7.5 回金
当结果不稳、证据不足、冲突过高时，回到热层、温层、冷层、原始镜像重新校准。

对应正式模块：**回源层 + 审计层**

必须钉死：

**吸金不是点金。**
**点金不是沉金。**
**沉金不是供金。**
**供金不是回金。**

任何一环被省掉，系统都会退化成半套。

## 九、热层 / 温层 / 冷层的正确含义

## 9.1 冷层 Cold
### 定义
冷层是原始资产真身层与历史回源层。

### 存放内容
- PDF / DOCX / XLSX / MD / TXT
- 图片
- 音频
- 视频字幕结果
- 网页快照
- 原始抓取镜像
- 被替代但需保留的历史原件

### 作用
- 保存真身
- 支撑回源
- 支撑审计
- 作为最终证据底座

## 9.2 温层 Warm
### 定义
温层是元数据治理中枢层。

### 存放内容
- 文档记录
- 节点记录
- 作用域
- 状态
- 版本
- authority_score
- trust_score
- freshness_class
- supersedes / superseded_by
- storage_path / source_ref
- evidence_anchors
- 审计索引

### 作用
- 决定谁该被优先使用
- 决定谁该被压制
- 决定如何回源
- 决定生命周期迁移

## 9.3 热层 Hot
### 定义
热层是治理型向量候选层。

### 存放内容
- chunk 向量
- 知识节点向量
- 规则节点向量
- 摘要节点向量
- 证据节点向量
- 少量检索必要字段

### 作用
- 高速召回候选
- 提供供给前候选集
- 支撑热区构建

### 必须钉死
热层不是“全部资产总库”，而是**治理后长期留热的高价值向量指纹层**。

## 十、吸金接入的正式定义

本系统必须内建吸金能力。

## 10.1 吸金不是附加功能
它不是“后续增强功能”，而是供应链闭环的起点之一。

## 10.2 吸金支持的来源
1. 用户给定 URL
2. 用户给定关键词
3. 用户给定站点范围
4. 用户给定 API 数据源
5. 用户给定网页列表
6. 外部抓取任务

## 10.3 吸金输出对象
吸金输出不能直接跳过信炉。

必须至少输出：
- raw_asset
- source_type = web/api/search
- source_ref = URL/API 路径
- 抓取时间
- 抓取快照
- 正文文本
- trace_id

## 10.4 吸金与本地化的关系
“本地化系统”不等于“只吃本地文件”。

本地化的正确含义是：

**系统的治理、存储、审计与长期资产主权掌握在本地/私有侧。**

因此：
- 资料可以来自外部
- 但资产必须回到本地治理与沉淀

## 十一、四级知识结构

所有正式知识必须落入：
1. 行业级
2. 企业级
3. 项目级
4. 话题/任务级

### 硬约束
1. 不允许正式知识无层级
2. 不允许用模糊标签替代正式层级
3. session 不是正式知识层级
4. 会话碎片不得自动升级为长期正式知识
5. 当前话题级脏上下文不得污染项目级、企业级、行业级

## 十三、治理规则

## 13.1 质量规则
任何节点不得平权。

质量必须支持：
- authority_score
- verification_score
- reuse_score
- recency_score
- consistency_score
- conflict_penalty
- scope_stability_score
- mass_score
- mass_level

## 13.2 相位与作用域规则
- 不能只看 embedding 相似度
- 必须叠加作用域、时序、风险、权限、约束
- 当前作用域优先于跨域候选

## 13.3 版本与状态规则
状态必须支持：
- active
- superseded
- archived
- revoked
- expired

规则：
- 新版生效后，旧版不得与新版同权竞争
- revoked 原则上不进入正式供给
- archived 不进入默认热召回

## 13.4 供给规则
- 不直接把原始大块资料送给模型
- 只供给被治理后的信息
- 高风险或低证据场景必须支持回源

## 二十二、最终定义

龙码信息资产本地化系统不是单一知识库，不是单一向量模块，也不是单一聊天助手的附件。

它的正确总定义是：

**在龙码协议约束下，以吸金接入、本地准入、龙码信炉、四级知识结构、元数据治理、治理型向量知识库、供给、回源、审计、生命周期管理为完整闭环，把本地与外部原始资料转化为可沉淀、可治理、可供给、可回源、可审计的信息资产底座系统。**

一句话收束：

**它的任务不是把资料存起来，而是把资料炼成资产，再把资产稳定供给给 AI。**
