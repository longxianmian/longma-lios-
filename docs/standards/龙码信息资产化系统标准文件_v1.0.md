# 龙码信息资产化系统标准文件（校准版）

版本：P0 标准母文件  
状态：以原《龙码信息资产本地化系统技术开发文档》《龙码信息资产云化系统技术开发文档》《龙码信息资产本地化系统首版执行清单》为准重新校准  
定位：龙码协议约束下的信息供应链系统，不是普通知识库、普通 RAG、普通云盘、普通企业知识库。

---

## 一、总定义

龙码信息资产化系统，是在龙码协议约束下，把本地、云端、外部、多端、多来源原始资料，转化为可治理、可授权、可调用、可回源、可审计、可生命周期管理的信息资产底座系统。

它的任务不是“把资料存起来”，而是：

```text
把资料炼成资产，再把资产稳定供给给 AI、Agent、LIOS、行业系统、数字客服、数字销售、GEO 与第三方业务系统。
```

系统必须同时满足三项约束：

```text
龙码协议
  ↓
龙码信息供应链系统
  ↓
本地化 / 云化 / 混合化产品形态
```

---

## 二、系统不是哪个东西

本系统不是：

```text
普通本地文件搜索工具
普通云盘
普通企业知识库
普通 RAG 平台
单一向量数据库服务
单一聊天机器人后台
单一大模型替代品
LIOS 的附属插件
```

系统必须避免退化成：

```text
文件上传 + 切片 + embedding + cosine 检索
```

这种做法不符合龙码信息资产化系统定位。

## 三、系统在龙码体系中的位置

### 3.1 与龙码协议的关系

龙码协议定义底层治理约束：

```text
信息如何被治理
节点如何被描述
质量如何被计算
相位如何被匹配
证据如何被锚定
路径如何被裁决
调用如何被准入
结果如何被回源与审计
```

信息资产化系统不能绕开龙码协议另起字段、逻辑和裁决口径。

### 3.2 与龙码信息供应链系统的关系

龙码信息供应链系统是上位总架构。

信息资产化系统是它的本地化、云化、混合化产品实现。

必须继承完整闭环：

```text
原始资料接入
→ 龙码信炉
→ 四级知识结构
→ 元数据治理
→ 治理型向量知识库
→ 供给
→ 回源
→ 审计
→ 生命周期管理
```

### 3.3 与 LIOS 的关系

```text
龙码信息资产化系统：解决信息供给侧治理问题
LIOS：解决上层 AI 系统治理、逻辑裁决、执行门控、长链路抗漂移问题
```

信息资产化系统不是 LIOS 的附属插件，而是可独立商业化的现实入口产品。LIOS 调用它提供的治理后资产。

## 四、根本目标

本系统唯一任务：

```text
把本地、云端与外部资料持续炼成高质量信息资产，并稳定供给上层 AI 与业务系统使用。
```

必须实现 10 个目标：

```text
1. 承接本地与外部异构原始资料
2. 将原始资料转化为协议化资产对象
3. 建立统一四级知识结构
4. 建立热层 / 温层 / 冷层三位一体存储体系
5. 让信息只在被治理后进入模型视野
6. 支持回源、证据链、版本与状态追踪
7. 支持热温冷代谢与生命周期管理
8. 支持对 AI 助手、Agent、审查系统、行业系统稳定供给
9. 降低 AI 致幻、目标漂移、无依据输出问题
10. 成为个人版、企业版、云化版、混合版共同资产底座
```

## 五、主链：吸金 → 点金 → 沉金 → 供金 → 回金

这是信息资产化系统的核心主链。

```text
吸金 / 准入
→ 龙码信炉
→ 四级知识结构
→ 元数据治理
→ 治理型向量知识库
→ 供给
→ 回源
→ 审计
→ 生命周期管理
```

为便于工程理解，使用“金流”简称：

```text
吸金 → 点金 → 沉金 → 供金 → 回金
```

### 5.1 吸金

吸入原始资料原矿。

来源包括：

```text
本地文件
指定文件夹监听
手动上传
指定 URL
指定站点
搜索结果
API 数据源
企业系统数据
会话记录
代码仓库材料
日志与运维资料
录音转写结果
图片 OCR 结果
多媒体转写结果
```

输出对象：

```text
RawAsset / CloudRawAsset
source_type
source_ref
storage_path / storage_uri
sha256
mime_type
trace_id
ingest_status
```

吸金不能直接跳过信炉进入向量库。

### 5.2 点金

点金是龙码信炉的第一阶段，对原始资料进行炼化。

包括：

```text
文本抽取
清洗
切片
脱水
蒸发
去重
聚类
提纯
基础摘要
锚点提取
版本比对
协议对象生成
```

### 5.3 沉金

沉金是把炼化结果沉淀为长期可治理资产。

包括：

```text
知识单元
摘要单元
证据单元
锚点单元
版本镜像单元
上下文包索引单元
同步状态记录
协议节点
证据锚点
```

沉金对应：

```text
灵池沉淀 + 四级知识结构定位 + 元数据治理
```

### 5.4 供金

供金是把治理后的资产供给上层系统。

供给类型：

```text
检索包 RetrievalPack
推理包 ReasoningPack
执行包 ExecutionPack
回源包 SourcePack
审计包 AuditPack
Agent 可调用上下文包
```

供给必须带：

```text
source_ref
evidence_anchor
trace_id
scope
status
version
final_score
required_evidence
```

### 5.5 回金

回金是在证据不足、冲突过高、上下文不充分、权限不足、风险过高时，回到原始资料和证据链重新校准。

包括：

```text
回源冷层原文
回源网页快照
回源 API 原始响应
回源本地资产引用
回源历史版本镜像
请求本地补证
写入审计
```

必须钉死：

```text
吸金不是点金
点金不是沉金
沉金不是供金
供金不是回金
任何一环被省掉，系统都会退化成半套
```

## 六、龙码信炉标准定义

龙码信炉是信息资产化系统的信息炼化中枢，不得阉割。

标准定义：

```text
龙码信炉 = 点金坍塌 + 灵池沉淀 + 算子库 + 调度中枢 + 上下文打包 + 审计回放
```

### 6.1 点金坍塌

负责：

```text

清洗
切片
脱水
蒸发
去重

提纯
基础摘要生成
```

### 6.2 灵池沉淀

沉淀为：

```text
知识单元
摘要单元
证据单元
锚点单元
版本镜像单元
上下文包索引单元
```

### 6.3 算子库

至少包括：

```text

聚类算子
锚点提取算子
证据链构建算子

向量生成算子
上下文压缩算子
上下文打包算子

```

### 6.4 调度中枢

负责：

```text

状态推进
队列化任务

```

### 6.5 上下文打包

生成：


### 6.6 审计回放

记录：

## 八、热层 / 温层 / 冷层

### 8.1 冷层 Cold

冷层是原始资产真身层与历史回源层。

存放：

```text
PDF / DOCX / XLSX / MD / TXT
图片
音频
视频字幕结果
网页快照
原始抓取镜像
被替代但需保留的历史原件
```

作用：

```text
保存真身
支撑回源
支撑审计
作为最终证据底座
```

### 8.2 温层 Warm

温层是元数据治理中枢层。

存放：

```text
文档记录
节点记录
作用域
状态
版本
authority_score
trust_score
freshness_class
supersedes / superseded_by
storage_path / source_ref
evidence_anchors
审计索引
租户与权限字段
```

作用：

```text
决定谁该被优先使用
决定谁该被压制
决定如何回源
决定生命周期迁移
决定是否可供给
```

### 8.3 热层 Hot

热层是治理型向量候选层。

存放：

```text
chunk 向量
知识节点向量
规则节点向量
摘要节点向量
证据节点向量
少量检索必要字段
```

作用：

```text
高速召回候选
提供供给前候选集
支撑热区构建
```

硬约束：

```text
热层不是全部资产总库，而是治理后长期留热的高价值向量指纹层。
```

## 十、治理型向量知识库

治理型向量知识库不是普通向量库，而是治理后高价值候选热层。

标准检索流程：

```text
请求进入
→ 租户校验 / 本地权限校验
→ 作用域门控
→ Top-K 语义召回
→ 治理型精排
→ 证据锚点过滤
→ 回源条件判断
→ 输出供给包
→ 写审计
```

首版精排公式：

```text
final_score = semantic_score
            × authority_score
            × trust_score
            × freshness_weight
            × scope_weight
            × status_weight
            × permission_weight
```

本地首版简化五因子可用：

```text
final_score = semantic_score × authority_score × freshness_weight × scope_weight × status_weight
```

禁止：

```text
只按 cosine 返回最终结果
跳过权限校验
跨租户裸召回
revoked / expired 默认进入供给
把原始资料直接喂给模型
```

## 十三、本地与云端同步标准

云化系统不能强制吞掉所有本地原始资料。

必须支持三种运行模式：

```text
纯云端模式：资料、治理、向量、供给都在云端
本地优先模式：原始资料留本地，云端只保存摘要、锚点、索引或授权后的资产
混合模式：冷层在本地，温层 / 热层部分同步到云端
```

同步模式：

```text
full：完整资产同步
metadata：只同步文档记录、协议节点、摘要、锚点、source_ref、storage_pointer
index：只同步向量引用、chunk 摘要、scope、mass_score、evidence_anchor
source_proxy：云端只保存索引，本地作为回源代理
```

冲突处理：

```text
本地优先
云端优先
新版本优先
人工确认
双版本并存
```

同步核心字段：

```text
sync_id
local_asset_id
cloud_asset_id
sync_mode
sync_status
local_hash
cloud_hash
last_synced_at
conflict_status
```

## 十五、企业知识库标准化模块的位置

企业知识库标准化不是信息资产化系统的核心定义，也不是新建一套系统。

它只能是：

```text
龙码信息资产化系统之上的企业级知识生产模板 / 垂直适配模块 / 可售卖产品壳
```

它复用底层：

```text
吸金 / 准入
龙码信炉
四级知识结构
元数据治理
治理型向量知识库
供给
回源
审计
生命周期
```

它新增的是企业级知识标准模板：

```text
产品资料
服务流程
价格政策
常见问题
销售话术
案例资料
竞品对比
禁答边界
售后政策
转人工规则
```

这些内容进入系统时，必须被表示为：

```text
enterprise 级 DocumentRecord / ProtocolNode / EvidenceAnchor / Rule / Constraint
```

数字客服、数字销售、GEO 不能直接读取未治理资料，只能通过 SupplyPack 获取治理后资产。

## 十六、GEO 与数字客服/销售的位置

### 16.1 GEO

GEO 不是乱发内容，也不是黑帽投喂。

它是信息资产化系统供给出去的一个外部发布场景。

GEO 只能使用：

```text
已治理
可回源
可审计
状态可用
来源可信
企业确认
适合公开发布
```

的知识资产。

### 16.2 数字客服 / 数字销售

数字客服和数字销售不是直接接企业资料。

标准路径：

```text
企业资料 / 外部资料
→ 信息资产化系统炼化
→ SupplyPack
→ LIOS 治理判断
→ LLM 语言表达
→ 数字客服 / 数字销售输出
```

禁令：

```text
不得直接用 LLM 编造价格
不得直接用 LLM 编造案例
不得绕过禁答边界
不得绕过回源与审计
不得调用未确认或 revoked / expired 资产
```

## 十八、验收标准

### 18.1 原始资料准入验收

```text
放入 1 个 PDF，系统可自动归档到冷层
手动上传 1 个 DOCX，系统可登记为 raw_asset
导入 1 个 URL，系统可抓取正文并保存快照
所有入口都能写审计日志
```

### 18.2 信炉闭环验收

```text
文本可抽取
文本可切片
可生成 DocumentRecord
可生成 ProtocolNode
可生成 ChunkRecord
可生成 EvidenceAnchor
```

### 18.3 治理层验收

```text
所有正式知识都有四级作用域
所有正式知识都有治理字段
状态可更新
版本字段可写入
```

### 18.4 热层验收

```text
可生成向量
可写入 LanceDB / pgvector
向量与 metadata 绑定
检索结果可返回来源信息
```

### 18.5 检索验收

```text
可按作用域过滤
不是只按 cosine 裸排序
当前项目资料优先于跨项目资料
高权威正式文档优先于低权重碎片
```

### 18.6 回源验收

```text
能从检索结果定位回原始文件
能从网页结果定位回 URL 或快照
能返回路径 / 页码 / 片段信息
```

### 18.7 审计验收

```text
每次处理有 trace_id
可查准入日志
可查处理日志
可查检索日志
可查回源日志
```

### 18.8 系统定位验收

```text
不是普通本地文件搜索工具
不是单一向量库接入工具
不是没有吸金能力的半系统
不是没有温层治理的半系统
不是没有回源与审计的半系统
```

## 二十、最终收束

龙码信息资产化系统的准确总定义：

```text
在龙码协议约束下，以吸金接入、本地/云端准入、龙码信炉、四级知识结构、元数据治理、治理型向量知识库、供给、回源、审计、生命周期管理、本地云端同步、租户授权与商业化交付为完整闭环，把多来源原始资料转化为可沉淀、可治理、可授权、可供给、可回源、可审计的信息资产底座系统。
```

一句话：

```text
本系统不是把资料存起来，而是把资料炼成资产，再把资产稳定供给给 AI。
```

企业知识库、GEO、数字客服、数字销售，都只是这个底座之上的应用模块，不能反过来定义或削弱信息资产化系统主链。
