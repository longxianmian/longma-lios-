## 1. 工程目标

把标准文件中的"信息供应链系统"翻译成可运行的代码工程。

具体目标：

1. **可启动**：`docker compose up` 一键起本地服务
2. **可吸入**：能从本地文件、URL、API 等多源吸金
3. **可炼化**：信炉最小闭环跑通
4. **可治理**：四级知识结构 + 元数据治理字段全部落地
5. **可检索**：治理型向量库五因子精排可用
6. **可供给**：能向 LIOS 提供 SupplyPack
7. **可回源**：从检索结果能定位回原始文件
8. **可审计**：每个动作有 trace_id 留痕
9. **可隔离**：四级知识结构的分层隔离不被破坏（防串标基础）
10. **可扩展**：本地化版本能升级到云化版本

## 2. 双形态产品的工程关系

### 2.1 本地化与云化的关系

```
longma-info-asset-local（本地化）
  ├── 个人 / 单企业本地资料入口
  ├── 技术：Python + FastAPI + SQLite + LanceDB + 本地文件系统
  └── 部署：单机、Docker、本地服务器

longma-info-asset-cloud（云化）
  ├── 多企业 / 多项目 / 多端
  ├── 技术：Python + FastAPI + PostgreSQL + pgvector + OSS + Redis
  └── 部署：阿里云 / AWS / 私有云

两者关系：
  - 不是两套独立系统，是同一套核心抽象的两种部署形态
  - 核心代码（信炉算子、协议对象、四级结构、五因子精排）共享
  - 差异层：存储后端、租户机制、同步层
  - 本地化先行，云化基于本地化扩展
```

### 2.2 工程开发顺序

```
第一阶段：本地化版本（longma-info-asset-local）
  - 单租户简化版
  - 跑通完整主链（吸金 → 点金 → 沉金 → 供金 → 回金）
  - 验证信炉算子、四级结构、五因子精排

第二阶段：云化版本（longma-info-asset-cloud）
  - 基于本地化代码扩展
  - 加多租户、workspace、同步、计量、商业化交付
  - 替换存储后端（SQLite → PostgreSQL + OSS）

第三阶段：混合模式（双向同步）
  - 本地优先模式 + 云端协作
  - 三种运行模式（纯云端 / 本地优先 / 混合）的同步层落地
```

**P0 = 第一阶段（本地化）**。本文档主要规划这一阶段。

## 6. 与上下游系统的对接

### 6.1 资产化 → LIOS（向上对接）

资产化系统通过 SupplyPack 向 LIOS 供给治理后资产：

```
资产化 检索 API: POST /api/retrieve
    ↓
返回 SupplyPack
    ↓
LIOS 应用层（如标典）调用 LIOS API:
POST /lios/runtime/decide
    ↓ external_evidence 字段携带 SupplyPack 内容
LIOS EvidenceBinder 消费证据 → 治理裁决
```

**字段映射**（详见矩阵 v1.1.2 第 3.3.2 节）：
```
SupplyPack.items[].content       → external_evidence[].data
SupplyPack.items[].source_ref    → external_evidence[].source
SupplyPack.items[].evidence_anchor → external_evidence[].anchor
SupplyPack.items[].final_score   → external_evidence[].confidence
SupplyPack.trace_id              → LIOS Ledger trace_id（关联）
```

### 6.2 资产化 → 标典 / 天问 / 其他应用（向上对接）

应用层通过资产化的标准 API 完成：
```
POST /api/assets/upload          # 上传项目资料
POST /api/forge/jobs/{job_id}/run # 触发信炉炼化（云版）
POST /api/retrieve                # 检索
POST /api/supply/reasoning-pack   # 获取推理包
POST /api/source/locate           # 回源
GET  /api/audits/{audit_id}       # 审计
```

### 6.3 LIOS ↔ 资产化（不直接对接）

⚠️ 重要：**LIOS 不直接调用资产化系统的内部组件**。

```
✗ 错误：LIOS Kernel 直接调资产化系统的 retrieve API
✓ 正确：应用层（标典 / 天问）调资产化拿 SupplyPack，再传给 LIOS
```

理由：保持 LIOS 治理引擎纯净，不耦合资产化系统的演进。

## 7. 不在 v0.1 展开的内容（启动开发前 v0.2 细化）

### 7.1 信炉完整算子集

P2 只实现 5-7 个核心算子（extract / clean / split / dedup / summarize）。其余算子（脱水 / 蒸发 / 聚类 / 提纯 / 锚点提取 / 版本比对 / 上下文压缩 / 上下文打包 / 回源算子 / 审计记录算子等）启动开发前 v0.2 细化具体算法和实现路径。

### 7.2 五因子精排的具体权重

精排公式确定：
```
final_score = semantic × authority × freshness × scope × status
```

但每个因子的具体计算公式、权重系数、归一化方式留给 v0.2。

### 7.3 行业级资产升级的审定流程

⚠️ **关键**：行业级资产可跨租户共享，是防串标的最敏感点。

升级流程必须包含：
- 默认归类逻辑
- 升级触发条件
- 人工审核界面
- 审定后的版本控制
- 审定历史的不可篡改记录

具体设计留给 v0.2。

### 7.4 云化版本（longma-info-asset-cloud）的工程化

P0 只做本地化版本。云化版本基于本地化扩展，包括：
- 多租户机制（tenant_id / workspace_id 全字段化）
- PostgreSQL + pgvector 替换 SQLite + LanceDB
- OSS 替换本地文件系统
- Redis Queue / Celery / Dramatiq 替换 asyncio worker
- 同步层（full / metadata / index / source_proxy）
- 计量与配额
- 商业化交付层

具体施工路径留给云化阶段单独的工程化文档。

### 7.5 三种运行模式的同步层

纯云端 / 本地优先 / 混合模式的同步实现，留给云化阶段。

### 7.6 复用与隔离的完整规则集

矩阵 v1.1.2 第 3.2.7 节列了精简版 4 条硬约束。完整规则集（含 4 个执行细节、5 条铁律、跨企业相似度检测算法、投标代理隔离设计、差异化引导话术等）留给 v0.2 + 标典 v1.4 联动设计。

## 8. AI 协作硬约束（专门给 Claude Code 看）

### 8.1 总纪律

继承《龙码产品矩阵 v1.1.2》第 10.1 节："项目架构清晰、流程顺畅、代码优雅、运行稳定"。

### 8.2 资产化系统专属硬约束

1. **不允许把资产化系统做成普通 RAG**——必须有信炉、有四级结构、有治理元数据、有回源、有审计
2. **不允许 cosine 裸排序**——必须用五因子精排
3. **不允许跨租户裸召回**——租户隔离硬约束在代码层强制
4. **不允许跳过信炉直接进向量库**——主链不可断
5. **不允许无 source_ref 供给**——所有结果必须可回源
6. **不允许无 trace_id 供给**——所有动作必须可审计
7. **不允许无四级作用域入库**——禁止"无主资产"
8. **不允许在 LIOS Kernel 里调资产化系统**——只能由应用层桥接
9. **不允许把吸金/点金/灵池等代码塞进 LIOS 仓库**——独立仓库
10. **不允许默认升级用户上传资料到行业级**——必须经审定（防串标根基）

### 8.3 主链完整性硬约束

```
吸金 → 点金 → 沉金 → 供金 → 回金

任何环节被绕过 → 系统退化为半套 → 拒绝
```

具体表现：
- 看到"先把文件丢向量库再说"——拒绝
- 看到"用户上传完直接 cosine 召回"——拒绝
- 看到"省掉审计回放节省性能"——拒绝
- 看到"五因子精排太复杂用单因子先跑"——可临时降级，但必须明确是降级状态

### 8.4 看到这些情况立刻停下询问

1. 跨租户复用企业级及以下资产
2. 默认把上传资料标为行业级
3. 把资产化系统做成 RAG SaaS
4. 把资产化系统做成云盘
5. 在 LIOS Kernel 里写资产化逻辑
6. 在资产化系统里写 LIOS 治理逻辑
7. 用爬虫主动抓取招投标公告平台（合规风险）
8. 跳过信炉算子集中的任何一环（主链不可断）

### 8.5 看到这些名字必须如何理解

| 看到的术语 | 必须理解为 |
|---|---|
| RAG | 标准 RAG，**资产化系统不是 RAG** |
| 知识库 | **应使用四级知识结构**（行业级/企业级/项目级/话题任务级）|
| 三级知识库 | 已废弃，应使用四级知识结构 |
| 灵池 | 龙码信炉的"沉淀"子组件 |
| 信炉 | 点金坍塌+灵池沉淀+算子库+调度+上下文打包+审计回放 |
| SupplyPack | 资产化向 LIOS 供给的标准对象 |
| 五因子精排 | semantic × authority × freshness × scope × status |
| 复用边界 | 行业级跨租户 + 企业级及以下隔离 |
| 冷层 / 温层 / 热层 | 原始真身 / 元数据治理 / 治理向量 |
