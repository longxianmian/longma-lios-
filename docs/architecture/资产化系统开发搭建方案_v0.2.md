### 1.2 LIOSGovernanceService.decide() 内部 10 步流程

`src/service/LIOSGovernanceService.ts:97-260` 完整流程:

```
DecideRequest (含 tenant_id / pre_extracted_claims / external_evidence / projection_snapshot)
  │
  ▼
Step 1: claim 提取 (ClaimExtractor, 优先用 pre_extracted_claims)
  │
  ▼
Step 2: 证据绑定 (EvidenceBinder)
  ├─ Step 2.1: 🔴 retrieveKBSnippets(tenant_id, message)  ← KB 召回升级 inquiry.* binding
  └─ Step 2.2: extractVerifierFromExternalEvidence  ← 升级 order.query 证据
  │
  ▼
Step 3: 构造 KernelInput (CandidatePackBuilder + tenant_policy 注入)
  │
  ▼
Step 4: LIKernel.decide (律 1 + 律 2)
  │
  ▼
Step 5: ActionResolver.resolve (compute, 不写 ledger)
  │
  ▼
Step 6: 🔴 retrieveKBSnippets (再次调) + fetchHistoryBrief  ← 给 generator 上下文
  │
  ▼
Step 7: augmentDecision* (基于 ExternalEvidence / 未知商品)
  │
  ▼
Step 8: BoundedLLMGenerator.generate (reply_draft)
  │
  ▼
Step 9: BoundsAuditor.audit (三层 + retry, 整体原子)
  │
  ▼
Step 10: 计算 LedgerPayload + 返回 DecideResult
```

**🔴 重点**: Step 2.1 + Step 6 都调 `retrieveKBSnippets(tenant_id, message)` (代码层面 2 次, 实际 KB 召回 1 次因 60s 缓存)。**这是资产化系统未来要接管的唯一入口**。

                AND is_indexed = TRUE

LIOS Step 2.2 extractVerifierFromExternalEvidence 已经支持这条路径!

LIOS 内部 retrieveKBSnippets() 何时废弃:
  1. 资产化 P5 完成 (供金 + 回源稳定)

## 2. LIOS ↔ 资产化系统适配方案

### 2.1 LIOS 调资产化的协议 (核心边界)

**标准文件 §3.3 + 施工路径 §6.3 锁定的硬约束**:

```
✗ 错误: LIOS Kernel 直接调资产化 retrieve API
✓ 正确: 应用层 (标典 / 天问) 调资产化拿 SupplyPack, 再传给 LIOS
```

**理由** (来自施工路径 §6.3):
> 保持 LIOS 治理引擎纯净, 不耦合资产化系统的演进。

**完整时序**:
```
[1] 用户消息 → 标典/天问应用层 (产品 P0)
[2] 应用层 → POST /api/retrieve (资产化系统)
       Body: { tenant_id, scope, query, requester='biaodian' }
[3] 资产化系统 → 内部信炉调度 → SupplyPack
       Return: { supply_id, trace_id, items[], audit_ref }
[4] 应用层 → POST /lios/runtime/decide (LIOS)
       Body: {
         tenant_id, source_app: 'biaodian',
         user_message,
         external_evidence: [
           { source: 'asset_system',
             type: 'supply_pack',
             data: SupplyPack.items[i],
             confidence: items[i].final_score,
             anchor: items[i].evidence_anchor }
         ],
         app_trace_id: SupplyPack.trace_id  // 跨系统串联
       }
[5] LIOS → service.decide() Step 2.2 消费 external_evidence
[6] LIOS → DecideResult + ledger 写入 (含 trace_link)
[7] 应用层 → 返回用户
```

### 2.2 资产化暴露给 LIOS 的接口 (SupplyPack 标准)

**标准文件 §14.5 SupplyPack 完整 schema**:

```json
{
  "supply_id": "SUP_001",
  "trace_id": "TRACE_001",
  "tenant_id": "TEN_001",
  "requester": "longma-agent|software-doctor|external-api|liOS|digital_sales",
  "verdict": "ALLOW|HOLD|REJECT",
  "scope": {},
  "items": [
    {
      "node_id": "CPN_001",
      "chunk_id": "CHK_001",
      "content": "......",
      "source_ref": "CRA_001#page=12",
      "evidence_anchor": "EA_001",
      "final_score": 0.87
    }
  ],
  "required_evidence": [],
  "audit_ref": "AUD_001"
}
```

**字段映射 (SupplyPack → LIOS external_evidence)** (施工路径 §6.1):
| SupplyPack | LIOS DecideRequest.external_evidence[i] |
|---|---|
| `items[i].content` | `data.content` |
| `items[i].source_ref` | `source` (字符串) |
| `items[i].evidence_anchor` | `anchor` |
| `items[i].final_score` | `confidence` |
| `trace_id` | (顶层) `app_trace_id` |
| `requester` | (顶层) `source_app` |

**LIOS 端不需要改动** (β-1 设计的 ExternalEvidence 接口已兼容此映射)。

### 2.3 token / tenant_id / trace_id 跨系统串联

**核心原则**: 两套系统**各自独立 tenant_id 命名空间**, 通过应用层桥接, **不直接共享 DB**。

#### tenant_id 同步策略
```
LIOS lios_tenants.tenant_id        资产化 tenants.tenant_id
       │                                      │
       └──────── 应用层 (标典/天问) 用同一 tenant_id 字符串 ───┘
                       (e.g. 'biaodian-demo')

LIOS 不知道资产化的 tenant 是否注册;
资产化不知道 LIOS 的 tenant_id 是否合法。
应用层负责保持一致 (注册时同时在两边登记)。
```

#### token 边界
```
LIOS lios_access_tokens (Bearer for /lios/runtime/decide)
   绑定 (tenant_id, source_app)

资产化 (本地化首版): 无 token (本地单租户)
资产化 (云化版):    自己的 access token, 与 LIOS token 完全独立

应用层 (标典/天问) 持有两套 token:
  - LIOS API token (调 /lios/runtime/decide)
  - 资产化 API token (调 /api/retrieve)
```

#### trace_id 跨系统串联
**依赖 LIOS β-3 commit 357fe16 已建好的机制**:
```
资产化生成 SupplyPack.trace_id (例: 'asset-trace-xxx')
应用层 → LIOS DecideRequest.app_trace_id = SupplyPack.trace_id
LIOS β-3 异步写 lios_trace_links:
  { lios_trace_id: 'lios-uuid-yyy',
    app_trace_id:  'asset-trace-xxx',
    source_app:    'biaodian',
    tenant_id:     'biaodian-demo' }

跨系统排错: SELECT lios_trace_links WHERE app_trace_id = 'asset-trace-xxx'
            → 拿到 lios_trace_id → 关联 LIOS 决策日志
            → 也可拿到资产化的 audit_ref (反查资产化系统审计)
```

### 2.4 DB 共享策略候选

| 候选 | 描述 | 优 | 劣 | 推荐场景 |
|---|---|---|---|---|
| **A. 完全独立 DB** | LIOS 用 PG `lios_db`, 资产化用 PG `longma_asset_db` | 边界清晰 / 演进解耦 | 跨系统查询要走 API | **推荐**: 与 §3.3 "可独立商业化" 一致 |
| B. 共库 schema 隔离 | 同一个 PG, schema=`lios` vs schema=`asset` | 单 DB 运维 | schema 演进互相牵制 | 仅当资源紧张时 |
| C. 共库共 schema | 共用所有表 | 简单 | 强耦合, 违反 §3.3 | **不推荐** |

**我推荐候选 A** (独立 DB), 与标准文件 §3.3 "信息资产化系统不是 LIOS 的附属插件, 而是可独立商业化的现实入口产品" 一致。

资产化 DB 设计参考 §11.4 本地化核心表:
```
raw_assets / documents / protocol_nodes / chunks / ingest_jobs / audit_logs
```

(本地化首版用 SQLite + LanceDB, 云化版用 PostgreSQL + LanceDB 替代)

### 4.2 资产化系统 P0 估算 (Claude Code 全程加速)

⚠️ **跟 LIOS v2.2 不同**:
- LIOS v2.2 是**增量改造** (在 v2.1 基础上)
- 资产化是**从零起** (无既有代码可改)
- 信炉 15 算子 + 6 协议对象 + 16 个 API + 4 级结构 + 5 因子精排, **复杂度 3-5x v2.2**

| Phase | Task 数 | 估算工时 | 说明 |
|---|---|---|---|
| P0 项目骨架 | 6 | 4-6 hr | FastAPI / SQLite / LanceDB 套件首装 |
| P1 原始资料入口 | 6 | 6-8 hr | 多格式解析 + 抓取适配 |
| P2 信炉最小闭环 | 9 | 12-16 hr | **核心**: 7 个算子 + 主链 stub |
| P3 治理层 | 6 | 8-10 hr | 四级 + 五因子 |
| P4 热层与检索 | 6 | 10-14 hr | 三池 + retrieve + SupplyPack |
| P5 回源与审计 | 6 | 8-10 hr | 回源 + 完整证据链 |
| P6 生命周期 | 6 | 6-8 hr | 状态机 + 测试 + 文档 |
| **总** | **45 task** | **54-72 hr** | **不含 OI / 蓝图修订 buffer** |

**实际加 buffer (按 v2.2 OI 占 30% 估算)**:
- 实际工时: **70-95 hr**
- 全职 1 人 1.5-2 周 / Claude Code 加速 + 用户决议

⚠️ **与用户期望对比**: 用户原话 "33-50 小时 / 4-7 工作日" — 这个估算**只够 P0 + P1 (~10-14 hr) 加上启动 P2 部分**, **不能覆盖完整 P0-P6 16 周**。

**两种解读**:
- (a) 用户期望是 "P0 项目骨架 + P1 原始资料入口" (即施工路径文档说的 Week 1-4) → 33-50 hr 估算合理
- (b) 用户期望是"完整 P0-P6 都按 LIOS v2.2 加速节奏" → 不现实, 至少 70-95 hr

**我推荐 (a)**, 在汇报里点出这个解读差距, 用户拍板。
